# 构建你自己的 AI 智能体：设计空间指南

> 这不是编码教程。这是关于构建生产级 AI 智能体系统时**你必须做出的设计决策**的指南，源自对 Claude Code 的架构分析。

每个生产级编码智能体都必须回答相同的反复出现的设计问题。Claude Code 是一组答案。本指南映射决策空间，以便你做出自己的知情选择。

---

## 决策 1：推理放在哪里？

**问题：** 你在模型中放置多少决策权 vs. 在你的 harness 代码中？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **最小脚手架** | Claude Code（约 1.6% AI 逻辑） | 模型拥有最大自由度；harness 强制执行边界。押注模型能力随时间改进。 |
| **显式状态图** | LangGraph | 开发者控制流程；更容易调试和预测。但约束模型，需要随能力改进而更新。 |
| **重型规划脚手架** | Devin | 多步规划器 + 任务跟踪器。更可靠用于复杂工作流，但脚手架本身成为维护负担。 |

**来自 Claude Code 的关键洞察：** 当前沿模型能力趋同（前三名在 SWE-bench 上相差 1% 内），**运营 harness 成为差异化因素**，而不是模型或脚手架。投资确定性基础设施（上下文管理、安全、恢复）可能比添加规划约束产生更高的可靠性。

**问自己的问题：**
- 你目标模型的能力如何？更能力的模型需要更少的脚手架。
- 你的工作流需要多可预测？受监管领域可能需要显式图。
- 模型能力改进多快？如果模型超越它，重型脚手架成为技术债务。

## 决策 2：你的安全姿态是什么？

**问题：** 你如何防止智能体做有害的事情？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **拒绝优先与分层强制** | Claude Code（7 个独立层） | 非常安全，但可能产生批准疲劳（93% 的提示未经审查批准）。需要渐进式信任机制。 |
| **容器隔离** | SWE-Agent、OpenHands（Docker） | 强边界，但粗粒度。容器内一切都允许；容器外都不可达。 |
| **VCS 回滚** | Aider（基于 git） | 轻量级，但仅保护文件更改。不能防止网络请求、数据泄露或 shell 副作用。 |
| **仅批准** | 基本聊天机器人 | 简单，但在规模上行为不可靠。用户停止阅读提示。 |

**来自 Claude Code 的关键洞察：** 深度防御仅在安全层具有**独立故障模式**时有效。Claude Code 的层共享经济约束（token 成本）——超过 50 个子命令的命令完全绕过安全分析以防止 REPL 冻结。设计你的层以独立失败。

**关于批准疲劳的关键洞察：** 用户批准 93% 的权限提示。解决方案不是更多警告而是**重构边界**——沙箱和分类器为自主操作创建安全区域。

**问自己的问题：**
- 你的智能体可能做的最糟糕的事情是什么？（删除生产数据？发送电子邮件？泄露代码？）
- 你能用沙箱化来减少用户必须做的决策数量吗？
- 你的安全层共享故障模式吗（例如，都依赖 token 预算）？

## 决策 3：你如何管理上下文？

**问题：** 上下文窗口是有限的。你如何决定模型看到什么？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **渐进式压缩管道** | Claude Code（5 层） | 为最长时间保留最多信息。复杂实现和调试。压缩对用户不可见。 |
| **简单截断** | 许多基本智能体 | 易于实现。但可能丢失关键的早期上下文。 |
| **滑动窗口** | 一些聊天应用程序 | 可预测的行为。但对重要的内容没有语义感知。 |
| **RAG（检索增强生成）** | 一些 IDE 集成 | 可以访问整个代码库。但检索质量是瓶颈，检索的块可能缺乏周围上下文。 |
| **单一摘要** | 一些智能体 | 一次摘要通过。但单一压缩可能丢失关键细节。 |

**来自 Claude Code 的关键洞察：** 上下文是塑造几乎所有其他架构决策的**绑定约束**。惰性加载、延迟工具模式、仅摘要子智能体返回和每个工具结果预算都存在，因为上下文是稀缺的。从第一天就为上下文稀缺设计。

**渐进式方法：** 首先应用最具破坏性最小的压缩。预算削减（便宜）→ 历史修剪（便宜）→ 缓存感知压缩（中等）→ 虚拟投影（中等）→ 完整摘要（昂贵，最后手段）。

**问自己的问题：**
- 你的上下文窗口大小是多少？这决定了你需要多积极的压缩。
- 你需要支持长会话（小时的工作）吗？单次截断无法存活。
- 你能分离"指导"上下文（指令）和"工作"上下文（对话）吗？

## 决策 4：你如何处理可扩展性？

**问题：** 外部工具、自定义指令和用户定制如何插入你的系统？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **渐进式上下文成本机制** | Claude Code（hooks=0，skills=低，plugins=中，MCP=高） | 不同扩展在不同成本。复杂管理，但可扩展。 |
| **单一统一 API** | 许多工具使用框架 | 简单理解。但每个扩展消耗上下文，限制可扩展性。 |
| **插件市场** | IDE 扩展 | 丰富的生态系统潜力。但质量控制和安全审查成为瓶颈。 |

**来自 Claude Code 的关键洞察：** 并非所有扩展都需要消耗上下文 token。Hooks（零成本）处理生命周期事件而不触及上下文窗口。Skills（低成本）仅在相关时注入。将高上下文成本机制（MCP）保留用于真正新的工具表面。

**三个注入点：** 每个智能体循环都有三个可以干预的地方：
1. **assemble()** -- 模型看到的内容（指令、工具模式）
2. **model()** -- 模型能触及的内容（可用工具）
3. **execute()** -- 操作是否/如何运行（权限门控、pre/post hooks）

**问自己的问题：**
- 你的智能体需要支持多少工具？更多工具 = 更多上下文压力。
- 你需要第三方扩展吗？为安全和质量控制做计划。
- 你能延迟工具模式加载直到模型实际需要工具吗？

## 决策 5：子智能体如何工作？

**问题：** 当智能体生成子任务时，它们共享上下文还是隔离运行？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **隔离上下文 + 摘要返回** | Claude Code（侧链转录稿） | 防止上下文爆炸（约 7 倍 token 成本）。但子智能体不能共享细粒度状态。 |
| **共享上下文** | 一些多智能体框架 | 完整信息共享。但上下文随 N 个智能体快速填充。 |
| **消息传递** | Actor 模型系统 | 清晰的边界。但需要显式协议设计。 |

**来自 Claude Code 的关键洞察：** 子智能体会话成本约是标准会话的 7 倍 token。只有摘要返回给父级——完整历史永远不会进入父级上下文。这对上下文保存至关重要。

**问自己的问题：**
- 你的子任务需要看到彼此的工作吗？
- 你如何防止 N 个子智能体消耗 N * context_window 个 token？
- 子智能体继承父级权限，还是建立自己的权限？

## 决策 6：会话如何持久化？

**问题：** 当会话结束时会发生什么？什么会延续？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **追加式 JSONL** | Claude Code | 可审计、可重建、简单。但查询能力弱。 |
| **数据库** | 一些企业智能体 | 丰富查询、快速查找。但增加基础设施依赖并降低透明度。 |
| **无状态** | 大多数聊天 API | 最简单。但没有恢复、没有分支、没有审计跟踪。 |

**来自 Claude Code 的关键洞察：** **恢复时绝不恢复权限。** 信任总是在当前会话中重新建立。安全状态不应跨会话边界隐式持久化。

**关于审计能力的关键洞察：** 追加式 JSONL 意味着每个事件是人类可读的、版本可控的，无需专业工具即可重建。查询能力的轻微损失值得透明度。

## 元模式：三个反复出现的设计承诺

在所有六个决策中，三个模式在 Claude Code 的架构中反复出现：

1. **渐进式分层优于单体机制** -- 安全、上下文和可扩展性都使用堆叠独立阶段而不是单一解决方案。

2. **追加式设计偏好审计能力超过查询能力** -- 一切都可以重建；没有任何东西被破坏性编辑。

3. **确定性 harness 内的模型判断** -- 模型自由决定；harness 强制执行边界。1.6%/98.4% 比例不是偶然的。
